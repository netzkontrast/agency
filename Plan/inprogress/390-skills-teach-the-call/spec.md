# Spec 390 — Skills teach the call (code-mode examples + wire-naming + schema shapes)

> Discovered by the **Agency Self-Teaching Loop**
> (`docs/loops/agency-self-teaching-loop.md`), Pass 1 OBSERVE. A fresh subagent
> drove 34 MCP invocations across 23 verbs/11 clusters using ONLY the committed
> `using-agency` skill, and the failures clustered not on *bootstrap* (which the
> skill teaches) but on **calling and chaining the verbs**.

## Problem (evidence, not hypothesis)

Three compounding gaps make a cold agent mis-call the MCP — each observed live:

1. **Wire-naming is silently inconsistent (C1).** Bare substrate tools
   (`agency_welcome`, `intent_bootstrap`, `agency_doctor`,
   `memory_graph_provenance`) sit beside prefixed `capability_<cap>_<verb>` verbs.
   Nothing teaches the rule, so a fresh agent guesses a plausible-but-nonexistent
   `manage.provenance` / `manage_provenance` and burns a call.
2. **`get_schema` hides nested object/array shapes (C2).** `detail="detailed"`
   renders `context: any[]` when the verb actually requires `[{id, text}]`; an
   output-only field (`gate_check.blocked_on`) reads like an input. Passing the
   wrong shape raised `unexpected_keyword_argument` and **aborted the entire
   `execute` block** (the documented partial-write hazard), losing the batch's
   later writes.
3. **The teaching surface is thin and partly un-generated (C7).** `using-agency`
   is a STATIC constant (`install.py::_USING_AGENCY_SKILL_MD`) — it never learned
   the naming rule. Per-cap skills DO autogenerate, but
   `disclosure.parse_module_skill` synthesizes only ONE example, for the
   alphabetically-first verb, with a fake `intent:abc` — no per-verb coverage and
   no chaining example. (Owner directive 2026-06-23: *every skill needs MCP call
   examples; the skills are mostly autogenerated, so extend the generator code.*)

Root-cause is uniform: **the generator does not teach the call**. Hand-editing 36
skills would drift (rule 2); the fix is in the GENERATOR + the substrate
`get_schema`.

## Approach (derive, don't duplicate — rule 2)

Three small, decidable extensions; no new dependency:

1. **`get_schema` renders nested shapes.** When a param is object- or array-typed,
   show its required keys (e.g. `context: [{id, text}]`) instead of `any[]`,
   derived from the verb's declared schema. This is the single highest-leverage
   change — it would have prevented two of the three hard errors.
2. **`parse_module_skill` emits a `## Calling these verbs` block** derived from the
   capability's known verbs: one `await call_tool("capability_<cap>_<verb>", {...})`
   line per verb (intent_id threaded) plus ONE chaining example that bootstraps an
   intent and chains two verbs. `emit_skill` surfaces the block in every
   `SKILL.md`. The example uses the REAL prefixed wire name (no `intent:abc`
   sentinel that reads as a literal).
3. **`using-agency` teaches the naming rule.** Extend `_USING_AGENCY_SKILL_MD`
   with: the explicit list of **bare substrate tools** vs the
   `capability_<cap>_<verb>` rule; "**`get_schema` before the first call** of an
   unfamiliar verb, with `detail='full'` for object params"; and the partial-write
   idempotency warning restated at point-of-use.

The capability surface is the source of truth; everything renders from the live
registry + declared schemas, so all ~36 skills regenerate at once.

## Acceptance (Gherkin — behaviour, not implementation; rule 7)

```gherkin
Scenario: get_schema shows a required nested object shape
  Given a verb whose parameter is an array of objects with required keys
  When I call get_schema for that verb at detail "detailed"
  Then the rendered parameter shows the required nested keys
  And it does not render the parameter as a bare "any[]"

Scenario: a generated skill teaches the call with correct wire names
  Given a capability with a derived SkillDoc
  When the install renders its SKILL.md
  Then the SKILL.md contains a code-mode call example for the capability
  And every example tool name is a live, resolvable wire name
  And it contains one chaining example that bootstraps an intent then chains a verb

Scenario: using-agency distinguishes bare substrate tools from capability verbs
  Given the generated using-agency skill
  Then it names the bare substrate tools explicitly
  And it states the capability_<cap>_<verb> rule for everything else
  And it tells the agent to get_schema an unfamiliar verb before the first call
```

## Decisions (WH(Y) — for ADR extraction)

- **D1 — Fix the generator + substrate, never the 36 skills by hand.** Editing
  each `SKILL.md` would re-drift on the next refactor (rule 2). WHY: the docstring
  / registry / declared schema is the single source; the rendering derives.
- **D2 — `get_schema` shape-rendering is the highest-leverage change.** It closes
  the block-aborting error class at the substrate, benefiting every client (MCP +
  CLI), not just the skill reader.
- **D3 — Examples use REAL prefixed wire names + a threaded intent_id**, never a
  `manage.provenance`-style dotted guess or a bare `intent:abc` sentinel — the
  example IS the naming lesson.
- **D4 — `using-agency` stays a curated constant** (not docstring-derived) because
  it is the cross-cutting meta-skill; it gains the naming rule as authored prose.
