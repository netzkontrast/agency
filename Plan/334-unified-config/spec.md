# Spec 334 — Unified `.agency/config.yaml` (works out of the box)

> A single, human-editable **`.agency/config.yaml`** holding **all** agency config
> — core knobs + every capability's keys, and **especially the frugal-discipline
> settings** (Spec 332) — **generated by `setup` + `agency install`/init** and
> **validated + repaired by `agency_doctor`**, with **env → config.yaml →
> built-in-default** precedence and an **extensible per-capability registry**.
> This is GOALS.md **Goal 9 criterion 5** made real: *"`agency install` +
> SessionStart yield a working substrate with zero manual steps, `agency_doctor`
> green or naming the gap."* It is the substrate **Spec 332 + 327 build on** — the
> frugal discipline can't "work out of the box" until config has a durable home.
> Provenance `intent:7509dac0`; `reflection:5c527d1e`.

## Why (evidence + doctrine)

Agency config is **scattered** today: env vars (`AGENCY_DB`, `AGENCY_EMBEDDER`,
`JULES_API_KEY`, `ANTHROPIC_API_KEY`, …), `_db_path.py`'s own resolution, and
per-capability files (`novel/config.py`, `music/config.py`). There is **no single
home, no generated default, no doctor validation** — so a fresh clone has nothing
to read or edit, and Spec 332's `frugal_level` would have nowhere durable to
persist. The out-of-the-box requirement (user, 2026-06-19) is: **after setup +
init, one annotated file lists every knob at a sensible default, and doctor
confirms it or names the gap.**

**Format — YAML, deliberately (the ladder applied).** JSON is stdlib (zero dep)
but **cannot carry comments**; a *generated, hand-editable* config whose whole job
is to be read and tuned needs inline explanation per key. So `.agency/config.yaml`
+ a declared **`pyyaml`** dependency (already present transitively; lightweight).
`# frugal: YAML over JSON — comments are the feature, not golf.`

## Design

### `agency/_config.py` — schema, resolver, registry (the critical path)

- **Resolver** — `config_get(key, *, engine=None)` resolves **env var →
  `.agency/config.yaml` → registered default** (preserves every current `AGENCY_*`
  override). Each key declares its env alias, type, default, and a one-line doc.
- **Registry (extensible — Goal 4)** — `register_config_section(name, keys)`.
  **Core** registers its section; **each capability registers its own** (the same
  reflection-discovery pattern that registers verbs), so the generated file is the
  **union of all live config = "all agency config values"** and stays complete as
  capabilities are added — no frozen audit to rot.
- **Writer** — `config_set(key, value)` persists to `.agency/config.yaml`
  (non-destructive: preserves user edits + comments via a round-tripping dump),
  the durable store Spec 332's `frugal_level(level)` writes through.

### The generated file (shape)

```yaml
# .agency/config.yaml — generated by `agency install`; safe to edit.
# Precedence: environment variable > this file > built-in default.
core:
  db_path: .agency/session.db        # env: AGENCY_DB
  embedder: tfidf                    # env: AGENCY_EMBEDDER  (tfidf|bge)
  token_backend: tiktoken            # env: AGENCY_TOKEN_BACKEND
frugal:                              # Spec 332 — the minimal-code discipline
  level: full                        # env: AGENCY_FRUGAL_LEVEL  (off|lite|full|ultra)
  stamp_every_verb: true             # M2 per-verb envelope stamp
secrets:                             # presence reported by doctor; values via env only
  jules_api_key: ${env:JULES_API_KEY}
  anthropic_api_key: ${env:ANTHROPIC_API_KEY}
# <capability sections appended here by their own register_config_section()>
```

Secrets are **never written as literals** — the file records the env reference;
doctor reports *presence*, not the value.

### Generation points (out-of-the-box — zero manual steps)

1. **`scripts/setup`** (Spec 282) — after install, scaffold `.agency/config.yaml`
   if missing (the dev path).
2. **`agency install`** (Spec 065) / **`--scaffold`** — write/repair the file
   (the consumer path).
3. **SessionStart hook** (Spec 062) — ensure it exists every session; **add any
   missing registered keys non-destructively** (new capability ⇒ new keys appear,
   user edits + comments preserved). Merge, never clobber.

### `agency_doctor` integration (validate + repair)

- **Report** — a `config` block: per key, the **resolved value + its source**
  (`env` / `file` / `default`), so a surprised user sees *why* a value is what it
  is.
- **Validate** — unknown keys, invalid values (bad enum/type), missing required →
  surfaced in `next_steps` (doctor green or **names the gap**, Goal 9).
- **Repair** — `agency_doctor --write-config` (or `agency install --write-config`)
  adds missing keys at defaults, non-destructively.

## Slices (TDD)

1. **Resolver + registry + schema.** `_config.py`: `config_get/set`,
   `register_config_section`, core section, env→file→default precedence.
2. **Annotated scaffold generator.** `config_scaffold(path)` emits the commented
   file with every registered key at its default; idempotent + non-destructive
   merge of missing keys.
3. **Wire setup + install + SessionStart.** Generate/repair at all three points.
4. **Doctor report + validate + `--write-config`.**
5. **Capability registry proof.** `frugal`/`core` register; migrate `novel`/`music`
   config onto `register_config_section` as the open-set proof.

## Acceptance criteria

- **C1 (out of the box)** — after `agency install` (or `scripts/setup`),
  `.agency/config.yaml` exists with **every registered key** at a sensible default
  + a per-key comment; frugal `level: full`.
- **C2 (precedence)** — `config_get` returns env over file over default; an
  `AGENCY_*` override still wins (back-compat).
- **C3 (extensible = "all values")** — a capability's `register_config_section`
  causes its keys to appear in the generated file with no edit elsewhere (Goal 4).
- **C4 (non-destructive)** — re-scaffold / SessionStart repair preserves user edits
  + comments and only **adds** missing keys; secrets never written literal.
- **C5 (doctor)** — `agency_doctor` reports each key's resolved value + source,
  flags invalid/unknown/missing in `next_steps`, and `--write-config` repairs.
- **C6 (substrate)** — Spec 332 `frugal_level` reads/writes the `frugal:` section
  through `_config`; Spec 333 reads any installer toggles from it.

## Acceptance scenarios (Gherkin sketch)

```gherkin
Scenario: install generates the annotated config
  When I run "agency install" in a clean project
  Then .agency/config.yaml exists
  And it contains a "frugal: level: full" key with a comment
  And every registered config section is present

Scenario: env overrides the file
  Given .agency/config.yaml sets frugal.level "lite"
  And AGENCY_FRUGAL_LEVEL is "ultra"
  When config_get("frugal.level") resolves
  Then it returns "ultra" (env wins) with source "env"

Scenario: SessionStart adds a new capability's keys without clobbering
  Given a user-edited .agency/config.yaml with a custom comment
  And a newly installed capability registers a config section
  When the SessionStart repair runs
  Then the new keys are appended at defaults
  And the user's edits and comment are preserved

Scenario: doctor names a bad value
  Given .agency/config.yaml sets frugal.level "turbo"
  When agency_doctor runs
  Then it reports frugal.level invalid in next_steps
  And suggests --write-config

Scenario: secrets are never written as literals
  When the config is scaffolded
  Then secrets keys reference ${env:...}, not the value
```
