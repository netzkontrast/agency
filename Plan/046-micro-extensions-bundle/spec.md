# Spec 046 — Mikro-Extensions Bundle

## Why

Six small extensions surfaced from the deep-dive analysis (PR #17
thread, second-round subagent reports). None of them deserves its own
spec — each is a 1-day change — but bundling them lets one PR carry
the "polish" pass that complements the bigger capabilities
(`analyze` Spec 042, `document` Spec 043, `research` Spec 044).

The six extensions are:

| # | Source | Lands as | LOC |
|---|---|---|---|
| **F-A** | Superpowers splits `code-review` into `requesting-` and `receiving-` (one is dispatch, one is rigor against performative agreement) | Split agency `code-review` skill into 2 | ~120 |
| **F-B** | Superpowers `brainstorming` ships a browser-based visual-companion (HTML/JS/sh server) for mockups | Port the companion behind a `[brainstorm-visual]` extra | ~200 |
| **F-C** | SC `sc-git` generates conventional-commit messages from change analysis | New `branch.commit_smart` verb composing diff analysis + template | ~80 |
| **F-D** | SC `sc-estimate` provides multi-persona estimation with confidence intervals | New `develop.estimate` verb (decidable inputs only — LOC, file count, test count) | ~60 |
| **F-E** | SC ships `MODE_Token_Efficiency.md` as a behavioural overlay | Codify as a `token-efficiency` SKILL.md citing Spec 023 | ~80 |
| **F-F** | Superpowers `working-with-claude-code/scripts/update_docs.js` autosyncs from docs.claude.com via llms.txt | Port as `skills/plugin-development/scripts/update_docs.py` | ~100 |

Total: ~640 LOC across 6 small pieces. The bundling reduces PR cost.

- [ ] `branch.commit_smart(intent_id: str, files: list[str] = None,

- [ ] Returns `{sha: str, message: str, files: list[str]}`.

### F-D — develop.estimate

- [ ] `develop.estimate(scope: dict, intent_id: str) -> dict` verb
  added (role `transform`).
- [ ] `scope` input: `{loc: int, files: int, tests: int, axes:
  list[str]}`.
- [ ] Returns `{wall_clock_min: int, jules_sessions: int, confidence:
  "low"|"medium"|"high", rationale: str, signals_fired: list[str]}`.
- [ ] Estimation rules (decidable, NOT prose):
  - Base: `wall_clock_min = max(15, loc/40 + files*5 + tests*8)`.
  - +50% if `axes` contains `"security"` or `"performance"`.

  team velocity isn't part of the model (those are out-of-band).
- [ ] Test: `tests/test_develop_estimate.py` covers the boundary cases
  (confidence transitions, jules_sessions threshold).

## Design

### F-A: why split code-review

The Superpowers two-skill model separates **dispatch** (you want
review) from **rigor against performative agreement** (you received
review). Today agency's single `code-review` skill conflates them:
neither side gets full attention. The split costs no functionality and
gains the receiving-side discipline — which is the single most
common rationalization site ("the reviewer suggested X, so I'll do X"
without verifying X is correct or YAGNI).

### F-B: why ship visual-companion despite Node dep

Brainstorming about UI/UX is the textbook case where text-only fails.
Superpowers' companion is a small browser-served HTML page that lets
the user see mockups, diagrams, and comparisons rendered. Agency's
brainstorming SKILL.md (already loaded) cites the companion-offer
block — but no companion exists in agency today. This spec ports it
verbatim (Node dep included, behind an optional extra), preserving
the per-question decision discipline (the companion is OFFERED, not
default).

### F-C: smart-commit IS composition, not generation

Conventional commit messages have a deterministic structure
(`type(scope): subject\n\nbody`). The "smart" in `commit_smart` is
**deriving the type/scope/subject from diff stats**, not generating
prose with an LLM. Template + decidable rules. The result is
predictable and reviewable.

### F-D: estimate IS a rule table, not personas

SC's `sc-estimate` invokes architect + performance + pm personas. The
result is opinionated prose. Agency replaces this with a **decidable
rule table** producing a structured payload. The user can argue with
the rule (and patch it) but can't argue with persona-prose. This is
the same agency-vs-SC pattern that informs `analyze` (Spec 042).

### F-E: token-efficiency as a meta-discipline

SC ships `MODE_Token_Efficiency.md` as a behavioural overlay. Agency
has the substrate (Spec 023) but no discipline skill. This spec adds
the skill so the discipline is invoke-able alongside the others. A
walker drafting a new verb invokes `token-efficiency` to remind itself
of the five rules before writing the return shape.

### F-F: doc autosync — manual fire, not hook

Superpowers' `update_docs.js` is invoked manually. Agency follows the
same pattern (no SessionStart hook). Three reasons:

1. Upstream docs change infrequently; auto-fire wastes API calls.
2. Fetching during a session is slow; user shouldn't pay the wait.
3. Auto-fires lack `intent_id` — CLAUDE.md Rule #2 violation.

## Open Questions

1. **Visual-companion port quality.** Superpowers' companion is ~200
   LOC of HTML/JS/sh. Port verbatim, or rewrite to use the agency
   plugin's existing assets? Lean port-verbatim; no time saved by
   rewriting; provenance clean.
2. **Should commit_smart auto-fire on `branch.create_pr`?** Today
   `branch.create_pr` requires an existing commit. We could make it
   call `commit_smart` if there's no commit yet. Lean: explicit two-
   step (call `commit_smart` then `create_pr`). Cleaner provenance.
3. **estimate's `axes` parameter.** The +50% for security/performance
   is heuristic. Should the user provide a multiplier dict instead?
   Lean: rule-table for v1, dict-override for v2.
4. **token-efficiency skill cross-with-others.** Should every other
   agency skill add a "see also: token-efficiency" footer? Lean: yes
   for the heavy ones (analyze, document, research); no for the
   discipline ones (TDD, debugging).
5. **Doc-sync sha drift.** If upstream changes a doc, our reference
   diff is the whole file. Should we hash the upstream and skip the
   download when sha matches? Yes — easy win, ~5 LOC.
