# ADR-0003 — Domain plug-in contract (`Domain(ABC)` + `manifest.toml`)

## Context and Problem Statement

Today, each of the five handler-bearing domains (music, novel, jules, context, shared) and the one skill-only domain (agentic) has a different disk layout, a different registration pattern, and a different relationship to the central manifest. `music`'s 17 handlers register via a hand-rolled discovery walk; `jules`'s handlers register through a separate MCP server; `shared` has no register-point at all — its tools are added by hand. The central manifest is regenerated by ad-hoc scripts, not by a single contract.

Canvas §1 names the destination: *"5+1 domains plug in via `Domain(ABC)` with one `register(mcp)` method"*. Canvas §2 fixes the disk shape: `manifest.toml`, `handlers/`, `skills/`, `references/`, `state/`, `tests/`. Without that contract, ADR-0002 (single MCP server) and ADR-0009 (manifest generator, second wave) cannot share a common registration surface.

## Decision Drivers

- Canvas §2 ("The 5+1 domains") — fixes the per-domain disk layout verbatim.
- Canvas §1 pillar 4 ("One Manifest — generated from domain registrations").
- VOCABULARY §4 — the six-domain canon and the §4.1 Path A/B disambiguation table; folding requires a uniform host-side contract.
- Lesson 04 (`04-affects-list-incompleteness.md`) — `affects:` lists drift when there is no single discovery surface; the `manifest.toml`-driven approach eliminates that class of drift.
- ADR-0002 — one MCP server means one `mcp` instance to register against; the contract specifies the registration shape.

## Considered Options

### Option A — `Domain(ABC)` + `manifest.toml`, fixed five-child layout (RECOMMENDED)

Every domain ships a Python class subclassing `Domain(ABC)` with one required method, `register(mcp: FastMCP) -> None`. Alongside it lives `manifest.toml` declaring handlers, skills, references, state, and edges. Disk shape is fixed:

```
domains/<name>/
├── manifest.toml
├── handlers/
├── skills/
├── references/
├── state/
└── tests/
```

The central `bin/agency-build-manifest` walks `domains/*/manifest.toml`; the MCP boot loop walks `domains/*/__init__.py` and calls `register(mcp)`.

### Option B — Ad-hoc per-domain layout (status quo)

Each domain keeps its current shape. No `Domain(ABC)`. Registration paths differ per domain. The manifest generator walks heuristics per domain. Minimal migration cost today; high coordination cost forever.

### Option C — Single-tree skills only, no per-domain folders

Collapse to one flat tree (`skills/`, `handlers/`, `references/` at the repo root). Drop the per-domain subdirectory. Familiar and simple, but breaks VOCABULARY §4 (domains exist for a reason), erases the per-domain `state/` cache, and forfeits the ability to mark whole domains as deferred / unloaded.

## Decision Outcome

**Chosen: Option A — `Domain(ABC)` + `manifest.toml` + fixed five-child layout.**

- Base class `Domain(ABC)` lives at `servers/agency-mcp/src/agency_mcp/lib/domain.py` (per canvas §13 spec 002).
- Required method: `register(mcp: FastMCP) -> None`. Optional method: `health() -> ToolResult`.
- `manifest.toml` schema is `lib/schemas/manifest.schema.json` (ADR-0005's authoritative-schema pattern).
- The five children are mandatory directories (even if empty — empty `state/` is acceptable for tool-only domains like `context`).
- Cross-domain imports are forbidden in Python; cross-domain reference happens via `related:` frontmatter edges and Wave D graph queries (ADR-0008).

## Consequences

### Positive

- One discovery surface for `bin/agency-build-manifest`; lesson 04 drift becomes structurally impossible (the build script is the source).
- Adding a seventh domain is a mechanical operation — drop a directory in the shape, the MCP picks it up at boot.
- The `manifest.toml` per domain is a small, hand-auditable file; pull-request reviewers can see "what this domain claims to expose" without grepping handler bodies.
- Folder uniformity makes test layout uniform — `tests/` is always at `domains/<name>/tests/`.

### Negative

- Migration cost: all existing handlers must be moved into the new layout and wrapped in a `Domain` subclass. The five handler-bearing domains carry ~44 handlers between them today; that's 44 import-path changes plus the registration boilerplate.
- The fixed five-child layout has rough edges for the tool-only domains (`context`, `shared`) where `skills/` is empty by design; the cost is an empty directory + a one-line `.gitkeep`.
- `manifest.toml` becomes a second source of truth alongside the Python decorators (ADR-0005). Drift between the two becomes a new failure class — mitigated by the manifest-drift CI check (canvas §7), but a real cost.

### Neutral

- Per-domain Python namespaces stay self-contained — no global rename.
- The Python class boundary mirrors the disk boundary; there is no surprising path-vs-import mismatch.

## Falsifier triggers

- If a domain ships in `main` without a `manifest.toml` for two consecutive PR cycles, the contract is being routed around — successor ADR.
- If `Domain(ABC)` accumulates more than three required methods, the "one register-point" claim is falsified — successor revisits the contract size.
- If cross-domain Python imports re-appear in `main`, the no-direct-imports invariant is broken — successor reopens the graph-edge-only constraint.
