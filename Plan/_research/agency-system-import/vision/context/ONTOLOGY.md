# The Context Column: Ontology & Graph Store

This document defines the ontology structure, the underlying graph store, and the lifecycle operations (ingest and query) that enable name-driven discovery and relationship tracking across the agency matrix.

## Node Types

The graph is composed of strictly defined node types. Every node requires a unique identifier (typically a slug) and a `type` property.

- **Plugin:** A top-level container mapping to a physical repository or major sub-system.
- **Row:** A thematic slice of the matrix (e.g., `music`, `novel`, `jules`, `workflow`).
- **Cell:** The intersection of a column and a row (e.g., `agentic/music`, `context/novel`).
- **Skill:** An actionable capability defined in an `agentic` cell.
- **Handler:** The Python backend implementation backing a skill.
- **Tool:** An MCP-exposed function registered by a handler.
- **Spec:** A design or specification document governing a task or phase.
- **ADR:** An Architecture Decision Record.
- **Lesson:** A synthesized learning generated from a session.
- **Phase:** A step in a workflow pipeline defined in a `workflow` cell.
- **Gate:** A validation checkpoint within a phase.
- **Template:** A templating artifact used to generate files, located in a `context` cell.
- **Schema:** A validation definition for data or files, located in a `context` cell.
- **Artefact:** Any concrete output file generated by the system.
- **Session:** A logged execution instance of an agentic workflow.
- **Reference:** T3 contextual information, typically unstructured.

## Edge Types

Edges describe the relationships between nodes. They are directional and typed.

- **CONTAINS:** Represents hierarchical nesting.
  - `Plugin` → `Row`
  - `Row` → `Cell`
  - `Cell` → `Artefact` / `Template` / `Schema` / `Skill`
- **ADJACENT_TO:** Cross-cell semantic relationships, explicitly defined in frontmatter `related:` arrays.
- **PREREQUISITE_OF:** Execution order dependency. `Phase` → `Phase` or `Spec` → `Spec`.
- **SUPERSEDES:** Versioning and replacement. `Artefact` → `Artefact` or `ADR` → `ADR`.
- **USES_SCHEMA:** Validation dependency. `Artefact` → `Schema`.
- **USES_TEMPLATE:** Generation dependency. `Artefact` → `Template`.
- **DISPATCHES:** Orchestration trigger. `Skill` → `Phase` or cross-row invocation.
- **REVIEWED_BY:** QA tracking. `Spec` → `Session` (Review session).
- **PRODUCED_LESSON:** Insight extraction. `Session` → `Lesson`.
- **DERIVED_FROM:** Content lineage. Rendered output `Artefact` → Source `Artefact`.

## Backing Store

**Choice:** SQLite with the `age` (Apache AGE) extension or an equivalent lightweight Cypher-compatible layer over SQLite (e.g., `Kuzu` or a custom schema using recursive CTEs if strict zero-dependency is required, but a dedicated graph extension is preferred for query clarity). Given the python environment, `networkx` is acceptable for in-memory graph operations, but for persistent, easily queryable cross-session state, SQLite is the substrate.

**Justification:**
- **Zero Heavy Infrastructure:** Unlike Neo4j, SQLite is file-based and requires no separate daemon, honoring the "cold-start cost" and "dependency footprint" constraints.
- **Query Latency:** For the scale of agency configurations (thousands of nodes/edges, not billions), local SQLite provides sub-millisecond query latency.
- **Cypher Compatibility:** Allows declarative, name-driven graph traversals which are significantly easier for LLMs to generate and comprehend than complex SQL JOINs.
- **Write Latency:** UPSERT operations during the PostToolUse hook are nearly instantaneous.

## Ingest Path

Graph ingestion is fully automated to prevent drift.

1. **Trigger:** The `PostToolUse` hook fires after any successful write operation via the system's tools.
2. **Scan:** If the written file contains frontmatter (identified by `---` delimiters), the hook parses it.
3. **Upsert Node:** The hook UPSERTs a node into the graph. The primary key is the file path or designated `slug`. The `type` property is mapped from the frontmatter `type`.
4. **Upsert Edges:** The hook scans specific frontmatter arrays (`related:`, `depends_on:`, `supersedes:`, `superseded_by:`) and constructs the corresponding edges (`ADJACENT_TO`, `PREREQUISITE_OF`, `SUPERSEDES`). It also infers structural edges (`CONTAINS`) based on the file path.
5. **Commit:** The SQLite transaction is committed.

## Query Path

The `agentic` column's anchor-triad and the `/agency` router issue queries against the graph store to fulfill the matrix rules.

**Example 1: Name-driven discovery (Rule 3)**
Find all three cells for a given row name without substring search.
```cypher
MATCH (r:Row {name: "music"})-[:CONTAINS]->(c:Cell)
RETURN c.path, c.column
```

**Example 2: Row isomorphism / Template location (Rule 2)**
Find the template used to generate a specific artefact to understand its structure.
```cypher
MATCH (a:Artefact {slug: "chapter-1"})-[:USES_TEMPLATE]->(t:Template)
RETURN t.path, t.required_vars
```

**Example 3: Cross-column reference (Rule 1 / Rule 3)**
Find context schemas related to a specific workflow phase.
```cypher
MATCH (p:Phase {name: "character_design"})-[:ADJACENT_TO]->(s:Schema)
RETURN s.path, s.description
```
