# Overview — the 3×N matrix + result registry

> **STATUS — 2026-05-19**: design canonical; the matrix law and per-column ownership are stable. **Note on §2 (Result Registry)**: the *concept* is preserved — products live outside the system substrate — but the *mechanism* has been refined. Per `vision/03-architecture.md`, system metadata about an artifact lives as properties on an `Artefact` graph node in `context/_store/ontology.db`; the bytes live in user-owned storage via pluggable artifact drivers (`fs` / `repo` / `s3` / `http` / `drive`). NO `.meta.json` sidecar files are written next to user artifacts. The "sidecar" join-key role has moved into the graph node.

This document supersedes `vision/00-charter.md`. It distills the system's foundational law, resolves the output-artifact problem, defines minimal token-economy contracts, and pins domain assignments.

## 1. The Matrix Law

The system is a 3-column × N-row matrix.
- **Rows** are domains (music, novel, podcast, jules).
- **Columns** are structural components (`agentic`, `workflow`, `context`).

Every cell in a column shares an identical canonical shape. Knowing one `agentic` cell teaches you all of them.

- **`agentic` column**: The WHO and HOW. Owns intent-to-action routing (skills), tool definitions (MCP handlers), and cross-row dispatch (harness-in-harness). Does *not* own pipelines or state.
- **`workflow` column**: The WHEN. Owns process state, step-ordering (phases), handoffs, and policy (gates). Does *not* own Python tools or schemas.
- **`context` column**: The WHAT. Owns schemas, templates, unstructured references, and ontology graph integration. Does *not* own execution logic.

## 2. Q1 Resolution: The Result Registry

**Conclusion:** There is NO fourth peer column. Output artifacts belong in a per-row **artifact registry** (`result/<row>/`), explicitly decoupled from the system's working memory.

**Argument for Registry:**
The three matrix columns describe the *machine* (how the system works on itself). Output artifacts (audio files, book drafts, PDFs) are the *product*. Storing products in a peer column (like `context`) pollutes the graph substrate, causing massive token-budget regressions during context discovery. By isolating products into a queryable, addressable `result/` registry, the machine remains stateless and lightweight.

## 3. Q3: Domain Assignments & Edge Cases

Clean domain assignment pins data to its canonical home:
- **Agentic owns:** execution rules (skills, tool definitions, handler code).
- **Workflow owns:** process rules (pipelines, step ordering, gates, handoff envelopes).
- **Context owns:** memory (schemas, templates, ontology rules, the context graph).
- **Registry owns:** products (final artifacts, rendered files, join keys).

**Resolving Edge Cases (Provenance):**
*Example: "This audio came from this Suno-prompt at this generation-time."*
This provenance spans multiple concerns. Solution: The registry holds the raw artifact (`track.mp3`) and a minimal sidecar metadata envelope (`track.meta.json`). The sidecar holds **join keys** (e.g., `workflow_session_id`, `context_prompt_node_id`). The graph does *not* duplicate the MP3. The registry does *not* duplicate the prompt text. The context graph retains the event node; the registry points to it.

## 4. Code Mode: Minimal Export Contracts

To protect the token economy, each cell manifest exports *only* what cannot be derived from `(row, column)`. The central harness derives the fully qualified names at load time. What is not present cannot be queried via the context graph.

### Agentic Mode (Code Mode)
Exports slash-command triggers and MCP tool-group entry points.
```toml
# Derived: skill `<row>-producer`, tool namespace `mcp__<row>_analysis_*`
[skills]
exports = ["producer"]
[tools]
exports = ["analysis"]
```

### Context Mode
Exports entity schemas and template entry points. Traversal handles the rest.
```toml
# Derived: `tag:agency-system.local,2026:schema:<row>/Track`
[ontology]
node_types = ["Track"]
```

### Workflow Mode
Exports the pipeline entry-verbs.
```toml
# Derived: `mcp__<row>_scaffold_workflow`
[workflow]
entry_verbs = ["scaffold"]
```

## 5. Strict Cell Manifests (Canonical Music Row)

By enforcing strict prefix-omission, we achieve disk and context-token savings.

**Naive Form (Bloated):**
```toml
[routing]
namespace_prefix = "music_"
[skills]
exports = ["music-producer", "music-master"]
[tools]
exports = ["music-analysis", "music-synthesis"]
```
*Token Waste:* Repeats the row identifier unnecessarily. Wastes space in the `tools/list` payload.

**Strict Form (Token-Efficient):**
The harness derives skill-name = `{row}-{export}`, MCP namespace = `({row},{column})`, etc.

#### `agentic/music/manifest.toml`
```toml
[cell]
row = "music"
column = "agentic"

[skills]
exports = ["producer", "master"]

[tools]
exports = ["analysis", "synthesis", "review"]
```

#### `workflow/music/manifest.toml`
```toml
[cell]
row = "music"
column = "workflow"

[workflow]
entry_verbs = ["scaffold", "start", "resume"]

[[phases]]
id = "01"
path = "phases/01-first.md"

[[gates]]
id = "review"
path = "gates/review-gate.md"
blocks = "02"
```

#### `context/music/manifest.toml`
```toml
[cell]
row = "music"
column = "context"

[ontology]
node_types = ["Track", "Album", "LyricSheet"]

[templates]
"lyric_sheet" = { path = "templates/lyric_sheet.md.jinja" }

[schemas]
"track_meta" = { path = "schemas/track.schema.json" }
```
