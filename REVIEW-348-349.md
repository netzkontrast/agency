# Critical Engineering Review: Specs 348 (Frugal Port) & 349 (Event Bus)

**Target Branches:** `claude/stoic-goldberg-41j2yy`
**Scope:** Architecture, Doctrine Alignment, Gaps, Panel Review Verification

---

## 1. Architectural Soundness & Missed Gaps (Severity-Ranked)

### 🔴 SEVERITY 1 (Blocker): Spec 349 Graph-Write Amplification on High-Volume Events
**Gap:** Spec 349 §7 dictates that *every* delivery records an `Event` node with a `DELIVERED_TO` edge for provenance. However, the triggers include `hook:PreToolUse`, which fires on *every single raw tool call*.
**Why it fails:** This explicitly violates the **Spec 336** constraint ("do not graph-record high-volume events", "moved tool capture off the graph (95% bloat)"). While the *Frugal* subscription limits itself via `once_per`, *any* capability that registers an unbounded subscription to `PreToolUse` will instantly recreate the graph bloat Spec 336 was built to eliminate.
**Resolution needed:** The substrate event bus must explicitly forbid/drop graph-recording of high-volume events (like `PreToolUse`) at the engine level, regardless of subscriber configuration.

### 🔴 SEVERITY 1 (Blocker): Spec 349 `toolcalls.db` SQLite Concurrency on the Hot Path
**Gap:** Panel fold **M5** moved the `once_per` memoization to `.agency/toolcalls.db` to survive the fresh-process-per-hook model.
**Why it fails:** `PreToolUse` is a hot-path hook. Claude Code can and does fire multiple tool calls concurrently. Multiple `agency hook` processes trying to read/write a SQLite DB concurrently will hit `database is locked` (`OperationalError`) exceptions.
**Resolution needed:** The safety contract (§7) must explicitly define the concurrency handling for this SQLite hot path (e.g., WAL mode enforcement, short timeouts, and a strict "fail-open / silent bypass" degrade contract if the lock times out, so tool execution is never blocked).

### 🟠 SEVERITY 2 (Major): Spec 348 "Drop-In Bar" Violation (Coupling)
**Gap:** `CLAUDE.md` specifies the drop-in bar: *"If adding a capability needs an edit anywhere else, that coupling is the bug."*
**Why it fails:** Spec 348 graduates Frugal to `agency/capabilities/frugal/`, but relies on hardcoded checks in core substrate (`agency_welcome` and `engine.py` `_session_start_handler`) to enforce its "mandatory" invariants (Panel **B2**). If Frugal is a standard capability, core must not hardcode knowledge of it. If it is core, it should not pretend to be a decoupled capability.
**Resolution needed:** Either keep Frugal as a core discipline (not a capability folder), OR use Spec 349's event bus and the configurable gates to inject Frugal without modifying `_substrate_tools.py` and `engine.py`.

### 🟠 SEVERITY 2 (Major): Spec 349 Synchronous Drain Timeout Risk
**Gap:** Panel fold **M6** requires re-entrant enqueued emits to drain synchronously *before* the hook returns so `additionalContext` can be aggregated.
**Why it fails:** If external hooks (`run:` targets, §6) or complex capability verbs run during this synchronous drain, they can exceed the host's (Claude's) hook timeout limit, crashing the session.
**Resolution needed:** The synchronous event drain must have a strict, enforceable wall-clock budget. If the budget is exceeded, the drain must truncate, log a timeout warning, and return immediately.

### 🟡 SEVERITY 3 (Minor): Spec 348 Token Economy on `review` and `debt`
**Gap:** Panel **M2** rightly says "capture stays full within scope" to respect `CLAUDE.md #76`.
**Why it fails:** Graph capture should be full, but returning the raw unbounded ledger of a whole-repo `debt` scan directly to the LLM context as MCP text output violates the token budget (`CLAUDE.md` Spec 066/068).
**Resolution needed:** Separate the *graph capture* from the *MCP return*. The verb should return a token-bounded summary ("Recorded N findings to the graph... top 3 are..."), not the full ledger text.

### 🟡 SEVERITY 3 (Minor): Spec 349 Acceptance Tests Still Assert Implementation
**Gap:** Panel **M3** attempted to re-altitude tests to behaviour, but the Gherkin still checks internal DB state (`"an Event{name:'first_use_emit'} is recorded"`, `"search/manage lists the capability"`).
**Why it fails:** `CLAUDE.md #7` strictly forbids testing internal implementation.
**Resolution needed:** Rewrite the Gherkin to test observable LLM/prompt outcomes (e.g., "Then the context contains the Frugal snippet" and "Then the next identical tool call omits the snippet").

## 2. Doctrine Alignment & Panel Fold Review

- **Architectural Soundness (Substrate vs. Capability):** Spec 349's **B1** fold correctly aligns the event bus as a `substrate` tool, perfectly mirroring the Spec 338 Lifecycle precedent. This is fundamentally sound.
- **Single Source Rule:** Spec 348 upholds the single-source rule by defining the capability surface as an *exposer* of the Frugal machine owned by Spec 347, avoiding duplication. (Panel **M5** successfully mitigated this).
- **No Hardcoded Values:** Spec 348 successfully remedies the Ponytail hardcoded benchmark defect by moving to `benchmark.json` (Panel **m1**), which properly complies with the published-constant exception to `CLAUDE.md #8`.
- **Re-entrancy Contract:** Spec 349's **B2** and **M6** provide a functional logical mechanism (breadth-first, non-recursive drain), but it lacks the critical wall-clock timeout safety floor (see Severity 2 above).

---

## 3. Verdicts

### Spec 348 (Frugal Port): 🔴 REFINE
**Reason:** The capability vs. core tension is unresolved. It cannot be both a `capabilities/frugal/` drop-in and require bespoke core hook modifications. Additionally, the token economy for whole-repo scans must be strictly capped on the MCP return.

### Spec 349 (Event Bus): 🔴 REFINE
**Reason:** The architecture mandates graph-recording every delivery, which will immediately re-trigger the Spec 336 bloat disaster on the `PreToolUse` hot path. The reliance on concurrent SQLite writes in a multi-process hook environment without explicitly defining concurrency safety guarantees is a blocker.
