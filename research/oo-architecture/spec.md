# Spec 001 — Object-Oriented Architecture Redesign

## Why

The current implementation of the PR1 `agency` engine exposes an excellent external fastMCP surface (`search`, `get_schema`, `execute`). However, internal representations of operation returns and agent capability schemas lack definitive object-oriented models. Specifically:
- `ToolResult` envelopes are ad-hoc across capability outputs.
- Boundaries like `JulesBackend` and `VCSBackend` do not use a common protocol.
- `Skill` and `Phase` are built heavily on loosely typed dictionaries.

As synthesized from the robust architecture in `the-agency-system` (specifically `vendor/the-agency-system/Plan/decisions/0005-shared-toolresult-envelope.md:1`), unifying tool outputs allows strict token-budget checks and generic loop recovery. Further, `superclaude-framework/KNOWLEDGE.md` metrics indicate that explicit typing and pre-execution checks (PM Agent ROI) saves 25-250x tokens. By deeply analyzing the plugin inventories, we have modeled explicit `Phase` objects over nested dictionaries to enable dynamic validation.

**Full Functional Inventory Abstractions:**
1. **bitwize-music Pipeline Abstraction**: A single pipeline must orchestrate roughly 88 registered MCP tools reliably. This ranges from album/track CRUD (`find_album`, `create_album_structure`, `rename_album`), lyrics processing (`extract_distinctive_phrases`, `count_syllables`, `analyze_rhyme_scheme`), deep audio mastering/mix operations (`master_audio`, `polish_audio`, `analyze_audio`), promo generation (`generate_promo_videos`, `publish_sheet_music`), database sync operations (`db_sync_album`, `db_create_tweet`), to health checks (`diagnose`, `check_venv_health`). The sheer volume requires the `DriverRegistry` mapped below to distribute load across specific driver clusters (e.g. `AudioDriver`, `TextDriver`, `DBDriver`) rather than injecting specific functions.
2. **superpowers-marketplace Mapping**: The comprehensive scan of superpowers scripts, docs, and hooks implies a plugin engine must separate `Skill` behavior definitions from underlying script logic dynamically. The engine will adapt `Driver` capabilities dynamically based on `.toml` configurations mirroring `superpowers/plugins/` loading logic.
3. **SuperClaude Framework / Plugin**: The deep read of SuperClaude confirms the `install/setup machinery` requires unified `Boundary` protocols that abstract the underlying OS interaction during automated plugin setup.
