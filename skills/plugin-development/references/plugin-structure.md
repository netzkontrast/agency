# Plugin Structure — How the agency plugin uses it

> This file shows the Claude Code plugin contract through the lens of THIS
> repo. For the inner engine patterns (capability, skill-walker, driver,
> substrate tool) see [`agency-patterns.md`](agency-patterns.md).

## Critical rules (agency-flavored)

### 1. `.claude-plugin/` contains ONLY manifests

```
.claude-plugin/
├── plugin.json          # YES
└── marketplace.json     # YES
```

Never `.claude-plugin/skills/` or `.claude-plugin/commands/`. The host
looks for those at plugin root.

### 2. `${CLAUDE_PLUGIN_ROOT}` for paths in `.mcp.json` (NOT `${CLAUDE_PROJECT_DIR}`)

The actual agency `.mcp.json`:
```json
{
  "mcpServers": {
    "agency": {
      "command": "${CLAUDE_PLUGIN_ROOT}/bin/agency-mcp",
      "args": [],
      "env": {
        "PYTHONPATH":     "${CLAUDE_PLUGIN_ROOT}",
        "AGENCY_DB":      "${CLAUDE_PROJECT_DIR}/.agency/session.db",
        "JULES_API_KEY":  "${user_config.jules_api_key}"
      }
    }
  }
}
```

Note the deliberate mix:
- `command` and `PYTHONPATH` → `${CLAUDE_PLUGIN_ROOT}` (the plugin install).
- `AGENCY_DB` → `${CLAUDE_PROJECT_DIR}` (the user's working tree — each
  project gets its own bi-temporal graph at `.agency/session.db`).
- `JULES_API_KEY` → `${user_config.jules_api_key}` (a sensitive userConfig
  field, stored in the OS keychain).

This is the canonical pattern for an MCP plugin that produces per-project
state.

### 3. Never edit a regenerated file by hand

```
.claude-plugin/plugin.json         # regenerated
.claude-plugin/marketplace.json    # regenerated
.mcp.json                          # regenerated
commands/agency-*.md               # regenerated (from verb registry)
skills/help/SKILL.md               # regenerated (from capability map)
```

Edits get clobbered on the next `python -m agency.install`. To change
these files, edit the source:
- For `plugin.json` / `marketplace.json` / `.mcp.json` → `agency/install.py`.
- For slash-commands → verb docstrings + capability metadata.
- For `help/SKILL.md` → the engine's capability registry.

Then regen + commit the diff.

## Plugin Manifest — `plugin.json`

The actual agency `plugin.json`:
```json
{
  "name": "agency",
  "version": "0.1.0",
  "description": "Harness-in-harness agency engine: code-mode capabilities (macroskills) over one provenance graph.",
  "author": { "name": "netzkontrast" },
  "keywords": ["agency", "code-mode", "capabilities"],
  "skills":   "./skills/",
  "commands": "./commands/",
  "displayName": "Agency",
  "homepage":   "https://github.com/netzkontrast/agency",
  "repository": "https://github.com/netzkontrast/agency",
  "license": "MIT",
  "userConfig": {
    "jules_api_key": {
      "type": "string",
      "title": "Jules API Key",
      "description": "Optional. Needed only for the `jules` capability …",
      "sensitive": true
    }
  }
}
```

### Path-field semantics (the one that catches everyone)

- `skills`   → **extends** the default `skills/` directory.
- `commands` → **replaces** the default `commands/` directory.
- `agents`   → **replaces** the default `agents/` directory.

If we set `"commands": "./custom-commands/"`, the default `commands/` would
silently be hidden. We deliberately set both `"skills": "./skills/"` and
`"commands": "./commands/"` redundantly — clearer than relying on defaults.

### Minimal version (for a non-agency plugin)
```json
{
  "name": "my-plugin",
  "version": "1.0.0",
  "description": "Brief description",
  "author": { "name": "Your Name" }
}
```

## Development Marketplace — `.claude-plugin/marketplace.json`

The actual agency marketplace entry uses GitHub as `source`:
```json
{
  "name": "agency",
  "plugins": [
    {
      "name": "agency",
      "source": { "source": "github", "repo": "netzkontrast/agency" },
      "version": "0.1.0",
      "description": "..."
    }
  ]
}
```

### Marketplace `source` shapes

| Shape | Use for |
|---|---|
| `"./"` | Local dev marketplace (relative to marketplace.json) |
| `{"source": "github", "repo": "owner/name"}` | GitHub repo (agency's choice) |
| `{"source": "url", "url": "https://..."}` | Generic git URL |

> **Common mistake** — bare `"owner/name"` as `source` is invalid. Must be
> the object form.

**Local install for dev:**
```bash
/plugin marketplace add /path/to/agency
/plugin install agency@agency
```

## Component formats (with agency examples)

### Skills (`skills/<name>/SKILL.md`)

The canonical example IS this skill. Frontmatter rules:

```markdown
---
name: plugin-development                # kebab-case, required
description: Use when building or extending a Claude Code plugin — …
allowed-tools:
  - Read
  - Write
  - Edit
  - Bash
  - mcp__plugin_agency_agency__search   # MCP tools the skill needs
  - mcp__plugin_agency_agency__get_schema
  - mcp__plugin_agency_agency__execute
---

# Plugin Development
…
```

- `name` must be **kebab-case**.
- `description` must start with **"Use when…"** — the matcher hooks onto
  that phrase. The CSO linter (`capability_plugin_lint_skill`) enforces it.
- `allowed-tools` should list ONLY what the skill genuinely needs.

### Commands (`commands/<name>.md`)

Agency commands are regenerated — never authored by hand. The generator
writes them from each verb's docstring + metadata. Format:

```markdown
---
description: Brief description of what this command does
---

# Command Instructions
What Claude should do when this command is invoked.
```

If you author a non-agency plugin's commands by hand, this is the format.

### Hooks (`hooks/hooks.json`)

The agency plugin ships **no hooks** today. If we ever add one (e.g. a
SessionStart hook to bootstrap `.agency/`), the format is:

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          { "type": "command",
            "command": "\"${CLAUDE_PLUGIN_ROOT}/hooks/run-hook.cmd\" format.sh" }
        ]
      }
    ]
  }
}
```

> **WARNING — duplicate hooks file:** `hooks/hooks.json` is auto-loaded.
> Do NOT also reference it in `plugin.json`'s `hooks` field — that causes
> "Duplicate hooks file detected" errors.

**Cross-platform hooks** — use the polyglot `run-hook.cmd` wrapper. See
[`polyglot-hooks.md`](polyglot-hooks.md).

**Available hook events:** `PreToolUse`, `PostToolUse`, `UserPromptSubmit`,
`SessionStart`, `SessionEnd`, `Stop`, `SubagentStop`, `PreCompact`,
`Notification`.

### MCP servers

Two patterns. Agency uses the second.

**A) Inline in `plugin.json`:**
```json
{
  "mcpServers": {
    "server-name": {
      "command": "node",
      "args": ["${CLAUDE_PLUGIN_ROOT}/server/index.js"],
      "env": { "API_KEY": "${user_config.api_key}" }
    }
  }
}
```

**B) Separate `.mcp.json` file (agency's choice):**
See the actual `.mcp.json` above. Reason agency picked this: the manifest
file gets edited often (slash-command names from verb additions), but
`.mcp.json` is stable. Separating keeps the more-stable file diff-quiet.

### Agents (`agents/<name>.md`)

Agency ships **no agents** today (it has CapabilityBase verbs and skill
walkers instead). If we add one:

```markdown
---
description: What this agent specializes in
capabilities: ["capability1", "capability2"]
---

# Agent Name
When to invoke this specialized agent.
```

## File permissions

Scripts must be executable AND committed as executable:

```bash
chmod +x bin/agency-mcp bin/agency
git update-index --chmod=+x bin/agency-mcp bin/agency
```

The second line matters — `chmod` alone reverts on `git stash`/`checkout`.

Check agency's `bin/` is executable:
```bash
ls -l bin/        # all three should have +x
```

## How the regen flow works

`python -m agency.install` is the single source of truth. It:

1. **Imports the live engine** — `Engine(":memory:")`, fully discovered.
2. **Reads the verb registry** — every `@verb`-decorated method.
3. **Reads the capability ontologies** — node types, enums, skills.
4. **Templates:**
   - `.claude-plugin/plugin.json` — name, version, userConfig, paths.
   - `.claude-plugin/marketplace.json` — the marketplace entry.
   - `.mcp.json` — MCP server entrypoint.
   - `commands/agency-*.md` — one per macroskill, listing its verbs.
   - `skills/help/SKILL.md` — the capability map.
5. **Writes the files** — overwriting any hand-edits.
6. **Optionally scaffolds `.agency/`** (with `--scaffold-db`).

This violates "the graph is the store" (CLAUDE.md Rule #2) — files written
from engine state. Spec 017 tracks the refactor to graph-record-then-render.

## When you're building a non-agency plugin

Use this layout as a reference, drop the agency-specific bits:

```
my-plugin/
├── .claude-plugin/
│   ├── plugin.json
│   └── marketplace.json
├── skills/<name>/SKILL.md
├── commands/<name>.md
├── hooks/hooks.json + run-hook.cmd      # if needed
├── .mcp.json                            # if shipping an MCP server
└── README.md
```

The Superpowers `developing-claude-code-plugins` skill is the canonical
generic reference for that path. This skill assumes you're working on or
with the agency engine.
